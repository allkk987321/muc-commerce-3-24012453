# 
## **一、实训项目名称**
校园轻集市——AI辅助下的前端工程实践实训项目

---

## **二、实训项目定位**
“校园轻集市”是一个面向大学生校园生活服务场景的前端综合实训项目。项目围绕二手交易、失物招领、拼单互助、跑腿委托等校园高频需求，设计一个轻量化校园信息平台，使学生在相对真实的业务场景中完成需求分析、页面设计、接口调用、状态管理、数据展示、功能测试和项目答辩等完整实践过程。

本项目不是面向真实上线运营的校园交易产品，而是一个服务于前端工程能力训练的教学型项目。项目功能不追求产品级完整性，而强调通过典型业务场景串联 Vue3、Vue Router、Pinia、Axios、Element Plus、JSON Server、ECharts、Git 和 AI Coding 工具等技术与工具，使学生在“做中学、学中做”的过程中形成前端工程实践能力。

本项目采用“前端应用 + Mock API”的开发模式。前端使用 Vue3 生态完成单页应用开发，使用 JSON Server 模拟 RESTful API 服务，通过 db.json 提供资源数据，帮助学生在不引入真实后端语言和数据库的前提下，理解前后端分离、HTTP、JSON、RESTful、CRUD 和接口联调等关键概念。

## **三、实训设计理念**
本实训不采用传统“先完整讲技术、再集中做项目”的方式，而采用“项目驱动、AI辅助、任务推进、即时应用”的组织方式。

学生不需要在实训开始前精通所有技术，但应具备基本了解和使用意识。实训过程中，Vue3、Router、Pinia、Axios、JSON Server、ECharts、Git 等技术均作为项目开发工具出现，学生在完成具体功能时同步学习和巩固相关知识。

本实训特别强调 AI Coding 工具的合理使用。AI 不只是用于生成代码，而是贯穿需求分析、页面设计、代码生成、代码审查、错误排查、测试用例设计和项目优化等多个环节。学生需要学会将自然语言需求转化为可执行的开发任务，能够通过 Prompt 与 AI 协同完成初版代码生成，并对生成结果进行理解、修改、调试和验证。

因此，本项目的核心训练目标不是单纯“写出多少代码”，而是培养学生在 AI 时代的软件工程实践能力，包括需求拆解能力、工具使用能力、代码理解能力、问题定位能力、版本回退能力、功能测试能力和项目表达能力。

## **四、实训目标**
### **（一）知识目标**
1. 理解前端单页应用的基本结构，掌握 Vue3、Vite、Vue Router、Pinia 等现代前端工程化技术在项目中的基本作用。
2. 理解前后端分离开发的基本思想，掌握浏览器、前端应用、Axios、RESTful API 与 Mock API 服务之间的数据交互过程。
3. 理解 JSON Server 在教学项目中的定位，明确其本质是 REST API 模拟服务，而不是真实后端系统。
4. 理解本地状态管理与远程数据资源之间的区别，明确 Pinia、localStorage、JSON Server 各自适合承担的职责。
5. 理解 AI Coding 工具在软件开发流程中的作用，认识 AI 生成代码、人工审查代码、功能验证和版本回退之间的关系。

### **（二）技能目标**
1. 能够基于 Vue3、Vite 和 Element Plus 搭建结构清晰、界面友好的前端单页应用。
2. 能够使用 Vue Router 实现首页、列表页、详情页、发布页、消息页、个人中心和数据看板等页面之间的导航。
3. 能够使用 Axios 封装 API 请求，并完成对 JSON Server 提供的资源接口的访问。
4. 能够使用 JSON Server 设计和维护 db.json 数据文件，完成信息列表、信息发布、收藏、消息、状态更新等基础 CRUD 操作。
5. 能够使用 Pinia 管理用户信息、收藏状态、当前会话、消息提醒等前端状态。
6. 能够使用 Element Plus 完成表单、表格、卡片、抽屉、弹窗、标签、消息提示等常用交互界面开发。
7. 能够使用 ECharts 对校园轻集市中的信息类型、校区分布、状态统计等数据进行可视化展示。
8. 能够使用 Git 本地仓库对项目进行基本版本管理，在 AI 修改代码导致错误时能够通过提交记录进行回退。
9. 能够借助 AI Coding 工具完成需求拆解、页面初稿生成、代码优化建议、错误排查和测试用例设计，并对 AI 输出结果进行人工判断和修正。

### **（三）素养目标**
1. 培养学生从真实生活场景中发现需求、分析需求和表达需求的产品意识。
2. 培养学生基于业务流程进行页面组织、组件拆分和数据建模的工程思维。
3. 培养学生在人机协同开发环境下的代码审查意识、风险控制意识和调试能力。
4. 培养学生通过版本管理保护开发成果、降低试错成本的工程习惯。
5. 培养学生面向项目交付的测试意识、展示意识、沟通表达能力和团队协作能力。

## **六、项目简介**
“校园轻集市”是一个面向高校学生日常生活服务的轻量化校园信息平台。项目聚焦校园内常见的二手交易、失物招领、拼单互助、跑腿委托等场景，支持学生在同一平台中完成信息浏览、信息发布、收藏关注、模拟砍价、消息沟通、状态跟踪和数据查看等操作。

项目设计目标不是构建真实商业级交易系统，而是通过校园生活化业务场景，为前端实训提供一个足够真实、足够完整、又能在限定时间内完成的实践载体。

项目强调三个特点：

1. 真实场景：选题来源于大学生日常生活，学生容易理解业务背景和用户需求。
2. 轻量闭环：围绕信息发布、浏览、互动、收藏、状态更新形成基本业务闭环。
3. 工程训练：通过项目开发串联前端工程化、接口调用、状态管理、数据展示、AI 辅助开发和版本管理。

## **七、项目业务场景设计**
“校园轻集市”项目围绕大学生日常校园生活中的高频信息服务需求展开，选取二手交易、失物招领、拼单搭子、跑腿委托四类典型场景作为业务基础。这四类场景既具有较强的生活真实性，又能够覆盖前端项目开发中常见的信息发布、列表浏览、详情查看、搜索筛选、收藏关注、消息沟通、状态流转和数据统计等典型功能。

本项目并不追求对真实校园交易平台的完整复刻，而是将四类校园场景作为教学载体，通过有限而清晰的业务流程，引导学生理解如何从真实生活需求出发，逐步抽象为功能模块、页面结构、数据资源和前端交互逻辑。

### **（一）二手交易场景**
二手交易是校园轻集市中最核心的业务场景之一，主要面向学生之间闲置物品流转需求。例如，学生可以发布教材、数码产品、自行车、台灯、耳机、生活用品等闲置物品信息，其他学生可以浏览商品详情、收藏感兴趣的商品，并通过消息进行咨询或模拟砍价。

该场景主要体现“商品型信息”的业务特征。每条二手交易信息通常包括标题、分类、价格、校区、交易地点、图片、描述、发布人、发布时间、商品状态等内容。二手交易场景能够较好地训练学生对商品列表、详情展示、价格字段、收藏功能、砍价交互和状态更新等前端功能的实现能力。

在本项目中，二手交易不涉及真实支付、订单生成、物流配送和交易担保。模拟砍价仅作为教学功能，用于训练表单输入、消息生成和状态变化等前端交互逻辑。

### **（二）失物招领场景**
失物招领是校园中非常常见的信息互助场景，主要服务于学生发布遗失物品或拾获物品的信息。例如，学生可以发布丢失校园卡、钥匙、水杯、耳机、书包等信息，也可以发布捡到物品的招领信息。其他学生可以通过关键词、物品类型、校区、地点和状态进行检索，并通过消息联系发布人。

该场景主要体现“信息型互助”的业务特征。与二手交易不同，失物招领通常不以价格为核心，而更强调物品特征、丢失或拾获地点、时间、联系方式、状态更新等信息。因此，该场景有助于学生理解不同业务类型下表单字段和展示内容的差异。

在本项目中，失物招领主要用于训练信息发布、关键词搜索、标签分类、详情查看、状态更新和消息联系等功能。不涉及真实身份核验、物品归属证明和复杂审核流程。

### **（三）拼单搭子场景**
拼单搭子场景面向学生之间的临时性协作需求，例如奶茶拼单、外卖拼单、水果团购、网购凑单、学习搭子、运动搭子等。学生可以发布拼单或搭子信息，说明参与人数、截止时间、校区地点和具体要求，其他学生可以浏览信息、收藏关注，并通过消息进行沟通。

该场景主要体现“多人参与型信息”的业务特征。与二手交易和失物招领相比，拼单搭子更强调人数、时间、参与状态和沟通过程，适合用于训练列表筛选、状态展示、消息互动和简单数据统计等功能。

在本项目中，拼单搭子不实现真实报名、名额锁定、支付结算和复杂成员管理。系统只保留必要的参与人数、截止时间、状态标签和消息沟通等教学型字段，用于帮助学生理解多人协作类信息的前端表达方式。

### **（四）跑腿委托场景**
跑腿委托场景面向校园内临时任务协助需求，例如代取快递、代买物品、代送文件、代排队、代办校园事务等。学生可以发布跑腿任务，说明任务内容、任务地点、期望完成时间和酬劳金额，其他学生可以查看任务详情，并通过消息与发布人沟通。

该场景主要体现“任务型信息”的业务特征。跑腿委托与二手交易不同，它的核心不是物品流转，而是任务发布、任务响应和任务完成状态。因此，该场景适合用于训练任务信息展示、状态流转、消息联系和个人中心中“我的发布”管理等功能。

在本项目中，跑腿委托不实现真实接单系统、履约确认、支付结算、评价申诉和平台监管。项目只保留发布、浏览、沟通、状态更新等基本流程，用于体现任务型业务在前端项目中的表达方式。

### **（五）四类场景与功能模块的关系**
四类业务场景虽然在现实中具有不同的业务含义，但在本项目中可以统一抽象为“校园信息资源”。每条校园信息根据类型不同，呈现不同的字段重点和交互方式。二手交易强调价格与砍价，失物招领强调地点与状态，拼单搭子强调人数与截止时间，跑腿委托强调任务与完成情况。

因此，本项目采用统一的信息发布、统一的信息列表、统一的信息详情、统一的收藏关注、统一的消息沟通和统一的状态管理方式来承载四类场景。这种设计既能降低项目复杂度，又能帮助学生理解前端项目中“统一资源模型”和“差异化业务展示”之间的关系。

四类场景与主要功能模块之间的关系如下：

| **业务场景** | **业务特征** | **主要功能训练点** |
| --- | --- | --- |
| 二手交易 | 商品型信息，强调价格、图片、收藏、砍价和交易状态 | 商品列表、详情展示、价格字段、收藏、模拟砍价、状态更新 |
| 失物招领 | 信息型互助，强调物品特征、地点、时间和找回状态 | 信息发布、关键词搜索、标签分类、详情查看、状态更新 |
| 拼单搭子 | 多人参与型信息，强调人数、截止时间、参与状态和沟通 | 列表筛选、状态展示、消息互动、人数统计 |
| 跑腿委托 | 任务型信息，强调任务内容、地点、时间、酬劳和完成情况 | 任务发布、任务浏览、消息联系、状态流转、个人中心管理 |


通过上述设计，学生可以在同一套前端架构中体验不同业务场景的差异，理解如何将现实需求转化为页面、组件、数据字段和交互逻辑。同时，四类场景又被控制在统一资源模型之内，避免项目演变为复杂的校园综合服务平台，从而保证实训任务在限定周期内可完成、可展示、可评价。

## **八、技术路线**
本项目采用“Vue3 前端应用 + JSON Server Mock API”的技术路线。

前端部分主要包括：

1. Vue3：构建页面和组件。
2. Vite：完成项目创建、运行和构建。
3. Vue Router：实现单页应用路由管理。
4. Pinia：管理用户状态、收藏状态、消息状态等前端状态。
5. Axios：完成前端与 Mock API 服务之间的数据通信。
6. Element Plus：快速构建表单、表格、弹窗、抽屉、卡片等页面组件。
7. ECharts：完成数据看板和统计图表展示。
8. Git：完成本地版本管理和代码回退。
9. AI Coding 工具：辅助需求分析、代码生成、代码审查、错误排查和测试设计。

Mock API 部分主要包括：

1. JSON Server：提供 RESTful 风格的 Mock API。
2. db.json：保存 users、goods、favorites、conversations、messages、notices 等资源数据。
3. RESTful API：通过 GET、POST、PATCH、DELETE 等方法完成资源操作。

本项目不引入真实后端框架和数据库，不涉及真实用户认证、支付、订单、实时通信、权限控制和复杂业务规则。相关功能如消息回复、砍价反馈、信用分变化等均作为前端教学模拟处理。

## **九、功能范围与实现深度**
为保证项目在实训周期内可落地，本项目对功能深度进行明确控制。各功能模块以服务教学目标为主，不追求真实产品级复杂度。

### **（一）核心功能**
1. 本地身份创建  
学生进入系统后，可创建本地用户档案，包括昵称、学院、校区、身份角色等信息。该功能主要训练表单设计、本地状态管理和用户信息展示。
2. 今日集市首页  
展示最新信息、热门信息、我的收藏、未读消息和基础统计信息。该功能主要训练首页布局、卡片组件、数据聚合和页面导航。
3. 集市信息浏览  
展示二手、失物、拼单、跑腿等不同类型的信息，支持关键词搜索、类型筛选、校区筛选、状态筛选和简单排序。该功能主要训练列表渲染、查询条件管理和接口参数使用。
4. 信息详情查看  
以详情页或抽屉形式展示信息图片、标题、价格、地点、发布人、标签、描述、发布时间和状态。该功能主要训练动态路由、详情数据请求和组件复用。
5. 信息发布表单  
支持发布二手交易、失物招领、拼单互助、跑腿委托等信息，完成标题、类型、价格、地点、描述、标签等字段录入和基础校验。该功能主要训练 Element Plus 表单、表单校验、POST 请求和页面跳转。
6. 收藏与取消收藏  
支持对信息进行收藏、取消收藏，并在个人中心展示我的收藏。该功能主要训练 Pinia 状态管理、资源关系建模和前端状态同步。
7. 模拟砍价互动  
面向二手交易信息，用户可以输入出价，系统根据简单规则生成模拟回复，并写入消息记录。该功能主要训练表单交互、状态流转和消息生成，不涉及真实交易、订单和支付。
8. 消息中心  
展示会话列表和消息记录，支持发送消息和生成模拟回复，显示未读消息数量。该功能主要训练组件通信、状态更新和消息列表渲染，不涉及 WebSocket 和真实实时通信。
9. 个人中心  
展示用户资料、信用分、我的发布、我的收藏，并支持对自己发布的信息进行状态更新，如进行中、已完成、已关闭等。该功能主要训练数据筛选、状态更新和个人数据聚合。
10. 校园交易趋势看板  
使用 ECharts 展示信息类型占比、校区分布、状态统计等图表。该功能主要训练前端数据统计和可视化表达，统计结果由前端根据 JSON Server 返回的数据计算生成。

### **（二）辅助功能**
1. 本地数据管理  
支持示例数据恢复、数据重置或简单导出。该功能作为扩展任务处理，不作为核心验收要求。
2. 交易安全提醒  
展示校园线下交易注意事项，如公共地点交易、贵重物品验真、保护个人隐私、警惕异常低价等。该功能主要训练信息展示和产品体验设计，不涉及真实风控。

## **十、JSON Server 使用边界**
本项目使用 JSON Server 承担 Mock API 服务职责。JSON Server 负责提供资源数据和基础 CRUD 接口，使学生能够体验前后端分离项目中的接口调用过程。

JSON Server 适合承担以下任务：

1. 提供商品或信息列表数据。
2. 提供详情数据。
3. 保存新发布的信息。
4. 更新信息状态。
5. 保存收藏记录。
6. 保存会话与消息记录。
7. 为数据看板提供原始数据。

JSON Server 不承担以下任务：

1. 不实现真实用户登录认证。
2. 不实现复杂权限控制。
3. 不实现真实交易订单。
4. 不实现支付流程。
5. 不实现 WebSocket 实时聊天。
6. 不实现复杂推荐算法。
7. 不实现严格数据库约束和事务控制。
8. 不实现服务端复杂业务规则。

因此，本项目中的信用分、砍价回复、消息自动回复、未读数量、趋势统计等逻辑主要由前端完成。JSON Server 只负责保存和提供相关数据。

建议项目使用稳定版本 JSON Server，并在实训材料中统一安装方式和启动命令，避免不同版本导致接口行为不一致。

## **十一、AI 工具融入设计**
本项目将 AI 工具作为实训的重要支撑，而不是简单的代码生成器。AI 工具融入项目全流程。

推荐使用具备 AI Chat、代码生成、多文件编辑和 Agent 模式能力的 AI Coding 工具。教师可根据学校账号、网络环境和学生设备情况，选择 VS Code + GitHub Copilot、Cursor、Trae、opencode、通义灵码等工具作为实训支持环境。考虑到实训组织的一致性和教师演示的便利性，本实训以 opencode 作为主要推荐工具；学生也可根据自身设备条件、账号情况和使用习惯，选择其他具备类似 Agentic Coding 能力的工具完成项目开发。

### **（一）需求分析阶段**
学生使用 AI 辅助分析校园二手交易、失物招领、拼单互助、跑腿委托等场景，生成用户画像、用户故事、核心功能清单和页面结构草案。

训练重点：

1. 将生活化需求转化为软件功能。
2. 学会向 AI 清晰描述业务背景。
3. 学会判断 AI 输出内容是否合理。

### **（二）原型与页面设计阶段**
学生使用 AI 辅助生成页面布局建议、导航结构、表单字段、列表筛选条件和交互流程。

训练重点：

1. 从功能需求推导页面结构。
2. 理解信息层级和用户操作路径。
3. 对 AI 生成的页面方案进行人工筛选和优化。

### **（三）编码实现阶段**
学生使用 AI Coding 工具辅助生成 Vue3 组件、Element Plus 表单、Axios 请求、Pinia Store 和 ECharts 图表代码。

训练重点：

1. 能够编写有效 Prompt。
2. 能够读懂 AI 生成的代码。
3. 能够根据项目结构修改和整合代码。
4. 能够避免盲目复制代码。

### **（四）调试与优化阶段**
学生使用 AI 辅助分析报错信息、定位问题原因、优化代码结构和改进组件设计。

训练重点：

1. 学会描述错误现象。
2. 学会提供必要代码上下文。
3. 学会判断 AI 的修改建议是否安全。
4. 学会使用 Git 在修改失败后回退。

### **（五）测试与答辩阶段**
学生使用 AI 辅助生成测试用例、异常操作清单、答辩提纲和项目总结。

训练重点：

1. 建立功能测试意识。
2. 能够围绕核心功能展示项目成果。
3. 能够说明 AI 在项目开发中的具体作用和局限。

## **十二、Git 在本项目中的定位**
本项目中的 Git 不作为复杂团队协作工具讲授，而是作为 AI 辅助开发环境下的“工程保险丝”。

学生在使用 AI 修改代码前，应先进行一次本地提交。当 AI 生成的代码导致项目运行异常、结构混乱或功能失效时，学生可以通过 Git 查看历史记录并回退到稳定版本。

本项目要求学生掌握以下基本操作：

1. 初始化本地仓库。
2. 添加文件到暂存区。
3. 提交稳定版本。
4. 查看提交历史。
5. 回退到指定版本。

通过这种方式，学生可以理解版本管理在现代软件开发中的基本价值，尤其是在 AI 辅助编码场景下，形成“先保存、再尝试、可回退”的工程习惯。

## **十三、实训组织方式**
本实训建议采用“教师引导 + 学生实践 + AI辅助 + 阶段验收”的方式组织。

每个阶段围绕一个具体任务展开，教师先说明业务目标和技术目标，再给出必要的项目结构、接口示例和开发边界。学生在完成任务过程中，可以使用 AI 工具生成初版方案或代码，但必须进行人工理解、运行验证和必要修改。

教师应重点关注以下问题：

1. 学生是否理解当前功能对应的业务场景。
2. 学生是否理解代码在项目中的位置和作用。
3. 学生是否能够说清楚前端如何通过 Axios 调用 JSON Server。
4. 学生是否能够区分 Pinia、localStorage 和 JSON Server 的职责。
5. 学生是否能够使用 Git 保存阶段成果并进行回退。
6. 学生是否能够识别 AI 生成代码中的错误或不合理之处。

## **十五、考核**
本项目考核应突出过程性、工程性和展示性，不宜只看最终界面效果。

考核维度如下：

1. 需求理解与功能完成度  
重点考查学生是否理解校园轻集市业务场景，是否完成核心功能闭环。

重点考查项目结构是否清晰，组件划分是否合理，页面跳转、接口调用、状态管理是否能够正常运行。

5. Git 使用与风险控制能力  
重点考查学生是否能够进行本地版本提交，是否能在代码出错时通过版本记录进行回退。
6. 测试与项目展示能力  
重点考查学生是否完成基本测试，是否能够清楚展示项目功能、说明技术路线和总结开发过程。

## **十六、实训特色**
本项目具有以下特色：

1. 场景真实  
项目来源于大学生熟悉的校园生活服务场景，业务理解成本低，学生参与感强。
2. 技术完整  
项目覆盖 Vue3、Router、Pinia、Axios、Element Plus、JSON Server、ECharts、Git 等现代前端开发常用技术与工具。
3. 难度适中  
项目不引入真实后端、数据库、权限系统和支付系统，避免复杂环境配置影响前端实训主线。
4. 过渡自然  
通过 JSON Server 模拟后端接口，使学生能够提前理解前后端分离开发方式，为后续学习 FastAPI、Spring Boot、MySQL 等全栈技术奠定基础。
5. AI 融合  
项目将 AI 工具融入需求分析、页面设计、编码实现、调试优化、测试答辩全过程，符合当前软件开发工具链的发展趋势。
6. 工程导向  
项目强调 Git 版本管理、接口调用、状态管理、测试验证和项目交付，帮助学生形成基本工程意识。

## **十七、总体说明**
`**<font style="color:#AD1A2B;">“校园轻集市”</font>**`不是一个追求真实商业上线的完整产品，而是一个面向大学生前端能力培养的综合实训载体。项目通过轻量化校园业务场景，把前端页面开发、接口调用、状态管理、数据可视化、AI 辅助编码和版本管理整合到一个完整项目中。

在教学实施过程中，应始终坚持“功能服务教学、AI服务开发、工具服务工程”的原则。对于消息中心、模拟砍价、信用分、安全提醒等功能，应明确其教学模拟属性，控制实现深度，避免将项目误导为复杂产品开发。

通过本项目实训，学生不仅能够完成一个可运行、可展示的前端项目，更重要的是能够初步理解 AI 时代前端工程开发的基本流程和能力要求。
