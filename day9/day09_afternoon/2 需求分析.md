## **一、需求分析目标**
“校园轻集市”是一个面向大学生日常校园生活服务场景的前端实训项目。需求分析的目标不是设计一个完整商业级校园交易平台，而是围绕二手交易、失物招领、拼单搭子、跑腿委托四类高频校园场景，提炼适合前端实训的核心功能、页面结构、数据资源和交互流程。

本项目需求分析重点关注以下问题：

1. 大学生在校园生活中有哪些轻量化信息发布与互助需求。
2. 二手交易、失物招领、拼单搭子、跑腿委托四类场景分别对应哪些用户行为。
3. 这些用户行为可以抽象为哪些前端功能模块。
4. 不同业务场景之间有哪些共性需求和差异化需求。
5. 哪些功能需要完整实现，哪些功能只作为教学模拟。
6. 如何将需求转化为页面、组件、接口、状态和数据字段。

本项目采用教学型需求分析方式，以用户故事为主要表达方式，并结合功能需求、数据需求和验收标准，帮助学生理解“从真实场景到软件功能”的转化过程。

## **二、用户角色分析**
本项目不设计真实权限系统，也不区分复杂的管理员、商家、审核员等角色。为了降低项目复杂度，系统统一面向校园学生用户，但在需求分析中可以根据用户行为划分为不同角色视角。

### **（一）普通学生用户**
普通学生用户是系统的主要使用者，可以浏览校园轻集市中的各类信息，包括二手交易、失物招领、拼单搭子和跑腿委托。普通学生可以查看详情、收藏感兴趣的信息、与发布者进行消息沟通，也可以创建自己的本地用户档案。

### **（二）信息发布者**
信息发布者是发布校园信息的学生，可以发布闲置物品、失物招领、拼单搭子或跑腿委托信息。发布者可以在个人中心查看自己发布的信息，并根据实际情况更新信息状态，例如进行中、已完成、已关闭、已找回等。

### **（三）信息浏览者**
信息浏览者是主动寻找信息的学生，可以通过关键词、类型、校区、状态等条件筛选信息。信息浏览者关注的是快速找到自己需要的信息，例如想买二手教材、寻找丢失物品、加入拼单或接跑腿任务。

### **（四）项目演示者**
项目演示者主要用于实训展示和答辩场景，可以展示首页数据、功能流程、消息模拟、收藏状态、个人中心和趋势看板等内容。该角色不是系统真实用户，而是服务于教学验收和项目答辩。

## **三、业务场景分析**
“校园轻集市”围绕四类校园高频场景展开。这四类场景在现实生活中含义不同，但在系统中可以统一抽象为“校园信息资源”。每条校园信息都具有标题、类型、发布人、校区、地点、描述、状态、发布时间等基础字段，同时根据不同类型呈现差异化字段。

### **（一）二手交易场景**
二手交易场景主要解决学生闲置物品流转问题。例如，学生可以发布教材、耳机、自行车、台灯、电脑配件等闲置物品信息，其他学生可以浏览、收藏、查看详情，并通过消息进行咨询或模拟砍价。

该场景的核心特征是“商品型信息”。它强调价格、图片、成色、收藏、砍价和交易状态，适合训练商品列表展示、详情页、发布表单、收藏功能、消息沟通和状态更新。

### **（二）失物招领场景**
失物招领场景主要解决校园物品遗失与拾获互助问题。例如，学生可以发布丢失校园卡、钥匙、水杯、耳机等信息，也可以发布捡到物品的招领信息。其他学生可以通过关键词、地点、类型和状态查找相关信息。

该场景的核心特征是“信息型互助”。它不以价格为核心，而是强调物品特征、丢失或拾获地点、时间、联系方式和找回状态，适合训练分类信息发布、搜索筛选、标签展示和状态更新。

### **（三）拼单搭子场景**
拼单搭子场景主要解决学生之间临时性协作问题。例如，学生可以发布奶茶拼单、外卖拼单、水果团购、学习搭子、运动搭子等信息，其他学生可以查看要求、关注信息并通过消息沟通。

该场景的核心特征是“多人参与型信息”。它强调人数、截止时间、参与状态和消息互动，适合训练人数显示、截止时间展示、状态标签、列表筛选和简单数据统计。

### **（四）跑腿委托场景**
跑腿委托场景主要解决校园内临时任务协助问题。例如，学生可以发布代取快递、代买物品、代送文件、代排队等任务，其他学生可以查看任务详情并与发布者沟通。

该场景的核心特征是“任务型信息”。它强调任务内容、任务地点、期望完成时间、酬劳和任务状态，适合训练任务发布、任务浏览、消息联系、状态流转和个人中心管理。

## **四、用户故事**
用户故事用于描述不同用户在具体场景下希望完成的操作及其价值。本项目采用如下格式表达需求：

作为一名【用户角色】，我希望【完成某个操作】，以便【获得某种价值】。

### **（一）本地身份创建**
1. 作为一名首次进入系统的学生，我希望能够创建本地用户档案，以便在系统中发布信息、收藏信息和进行消息沟通。
2. 作为一名学生用户，我希望填写昵称、学院、校区和身份角色，以便系统能够在信息详情和个人中心中展示我的基本信息。
3. 作为一名项目演示者，我希望能够快速生成一个本地用户身份，以便在不接入真实登录系统的情况下完成项目演示。

### **（二）二手交易**
1. 作为一名有闲置物品的学生，我希望能够发布二手商品信息，以便让校内其他同学看到并联系我。
2. 作为一名想购买闲置物品的学生，我希望能够按关键词、价格、校区和状态筛选二手商品，以便快速找到合适的物品。
3. 作为一名浏览者，我希望能够查看商品详情、图片、价格、成色和发布人信息，以便判断是否值得联系。
4. 作为一名买家，我希望能够对二手商品进行模拟砍价，以便体验与卖家沟通价格的过程。
5. 作为一名发布者，我希望能够将商品状态更新为已完成或已关闭，以便其他用户了解当前交易状态。

### **（三）失物招领**
1. 作为一名丢失物品的学生，我希望能够发布失物信息，以便提高找回物品的可能性。
2. 作为一名捡到物品的学生，我希望能够发布招领信息，以便失主能够尽快联系我。
3. 作为一名寻找物品的学生，我希望能够按关键词、校区、地点和状态搜索失物招领信息，以便快速定位相关内容。
4. 作为一名信息发布者，我希望能够将失物状态更新为已找回或已认领，以便避免无效联系。

### **（四）拼单搭子**
1. 作为一名想发起拼单的学生，我希望能够发布拼单信息，以便找到同学一起参与。
2. 作为一名想寻找搭子的学生，我希望能够浏览拼单、学习、运动等搭子信息，以便找到合适的伙伴。
3. 作为一名浏览者，我希望能够看到拼单人数、截止时间和当前状态，以便判断是否还可以参与。
4. 作为一名感兴趣的学生，我希望能够收藏拼单信息或发送消息，以便后续继续关注。

### **（五）跑腿委托**
1. 作为一名有临时任务需求的学生，我希望能够发布跑腿委托，以便找到同学帮助完成任务。
2. 作为一名愿意接任务的学生，我希望能够查看任务内容、地点、时间和酬劳，以便判断是否适合自己。
3. 作为一名发布者，我希望能够与感兴趣的同学进行消息沟通，以便确认任务细节。
4. 作为一名发布者，我希望能够更新任务状态，以便标记任务进行中、已完成或已关闭。

### **（六）收藏与关注**
1. 作为一名浏览者，我希望能够收藏感兴趣的信息，以便后续快速查看。
2. 作为一名学生用户，我希望能够在个人中心查看我的收藏，以便集中管理关注的信息。
3. 作为一名浏览者，我希望能够取消收藏，以便清理不再关注的信息。

### **（七）消息中心**
1. 作为一名学生用户，我希望能够查看会话列表，以便知道自己与哪些发布者或浏览者有过沟通。
2. 作为一名学生用户，我希望能够查看聊天记录，以便了解之前的沟通内容。
3. 作为一名浏览者，我希望能够向发布者发送消息，以便咨询商品、失物、拼单或跑腿任务细节。
4. 作为一名项目演示者，我希望系统能够生成模拟回复，以便在没有真实后端和实时通信的情况下展示消息交互效果。

### **（八）个人中心**
1. 作为一名学生用户，我希望能够查看个人资料和信用分展示，以便了解自己的本地用户信息。
2. 作为一名信息发布者，我希望能够查看我的发布，以便管理自己发布的各类校园信息。
3. 作为一名信息发布者，我希望能够更新发布信息的状态，以便反映信息当前进展。
4. 作为一名学生用户，我希望能够查看我的收藏，以便快速回访感兴趣的信息。

### **（九）趋势看板**
1. 作为一名项目演示者，我希望能够查看不同类型信息的数量占比，以便展示平台数据情况。
2. 作为一名项目演示者，我希望能够查看不同校区的信息分布，以便展示校园信息活跃情况。
3. 作为一名项目演示者，我希望能够查看不同状态的信息统计，以便展示项目的数据可视化能力。

## **五、功能需求分析**
### **（一）本地身份创建**
本地身份创建用于模拟用户进入系统后的基础身份信息。系统不接入真实登录认证，不进行账号密码校验。用户只需要填写昵称、学院、校区和身份角色，即可生成本地用户档案。

主要功能包括：

1. 填写昵称、学院、校区、身份角色。
2. 保存本地用户信息。
3. 在首页、信息详情、消息中心和个人中心中使用该用户信息。
4. 展示本地信用分，信用分仅作为教学展示字段。

### **（二）今日集市首页**
今日集市首页用于展示项目整体入口和核心数据概览。首页应提供常用功能入口，并展示最新信息、热门信息、我的收藏、未读消息和基本统计数据。

主要功能包括：

1. 展示欢迎信息和当前用户信息。
2. 展示不同类型校园信息的快捷入口。
3. 展示最新发布信息。
4. 展示热门或高关注信息。
5. 展示我的收藏数量和未读消息数量。
6. 提供进入集市浏览、信息发布、消息中心、个人中心和趋势看板的入口。

### **（三）集市信息浏览**
集市信息浏览是项目的核心功能之一，用于展示二手交易、失物招领、拼单搭子和跑腿委托等校园信息。

主要功能包括：

1. 展示校园信息列表。
2. 支持按信息类型筛选。
3. 支持按关键词搜索。
4. 支持按校区筛选。
5. 支持按状态筛选。
6. 支持按热度、发布时间或价格进行简单排序。
7. 支持空数据状态提示。
8. 支持点击列表项进入详情查看。

### **（四）信息详情查看**
信息详情用于展示单条校园信息的完整内容。不同类型的信息具有不同字段重点，但详情页应保持统一结构。

主要功能包括：

1. 展示标题、类型、状态、校区、地点、发布时间、发布人信息。
2. 展示图片、标签和详细描述。
3. 二手交易展示价格、成色和模拟砍价入口。
4. 失物招领展示丢失或拾获地点、时间和物品特征。
5. 拼单搭子展示人数、截止时间和参与状态。
6. 跑腿委托展示任务地点、期望完成时间和酬劳。
7. 支持收藏或取消收藏。
8. 支持进入消息沟通。

### **（五）信息发布表单**
信息发布表单用于发布四类校园信息。系统应采用统一发布入口，根据用户选择的信息类型动态展示不同字段。

主要功能包括：

1. 选择信息类型：二手交易、失物招领、拼单搭子、跑腿委托。
2. 填写通用字段：标题、校区、地点、描述、标签。
3. 二手交易填写价格、成色、图片预设。
4. 失物招领填写丢失或拾获类型、地点、时间、物品特征。
5. 拼单搭子填写目标人数、当前人数、截止时间。
6. 跑腿委托填写任务内容、任务地点、期望完成时间、酬劳。
7. 进行基础表单校验。
8. 提交后将数据写入 JSON Server。
9. 发布成功后返回列表页或详情页。

### **（六）收藏与取消收藏**
收藏功能用于帮助用户保存感兴趣的信息。收藏不是复杂的关注系统，只用于训练资源关系和状态同步。

主要功能包括：

1. 在列表页或详情页收藏信息。
2. 再次点击可以取消收藏。
3. 收藏状态应在列表页、详情页和个人中心中保持一致。
4. 个人中心展示我的收藏列表。
5. 收藏数据保存到 Mock API 或前端状态中。

### **（七）模拟砍价互动**
模拟砍价只面向二手交易场景，用于训练表单输入、状态变化和消息生成。

主要功能包括：

1. 用户在二手商品详情中输入出价。
2. 系统根据简单规则生成卖家模拟回复。
3. 砍价内容写入消息记录。
4. 自动创建或更新对应会话。
5. 不生成真实订单。
6. 不处理支付、成交、评价和售后。

### **（八）消息中心**
消息中心用于模拟用户之间的沟通过程。系统不实现真实实时通信，不使用 WebSocket，只通过本地交互和 Mock 数据展示消息效果。

主要功能包括：

1. 展示会话列表。
2. 展示每个会话的最后一条消息。
3. 展示未读消息数量。
4. 点击会话查看聊天记录。
5. 支持发送文本消息。
6. 支持生成简单模拟回复。
7. 支持从详情页进入与发布者的会话。

### **（九）个人中心**
个人中心用于集中展示当前用户相关信息。

主要功能包括：

1. 展示本地用户档案。
2. 展示信用分字段。
3. 展示我的发布列表。
4. 展示我的收藏列表。
5. 支持对我的发布进行状态更新。
6. 支持进入信息详情查看。
7. 支持查看与自己相关的消息。

### **（十）趋势看板**
趋势看板用于展示项目的数据可视化效果。统计结果由前端基于 JSON Server 返回的数据计算生成。

主要功能包括：

1. 展示信息类型占比。
2. 展示校区分布。
3. 展示信息状态统计。
4. 展示最近发布趋势或热门信息排行。
5. 使用 ECharts 完成基础图表展示。

### **（十一）本地数据管理**
本地数据管理作为扩展功能，不作为核心验收要求。

可选功能包括：

1. 示例数据恢复。
2. 数据重置。
3. 简单数据导出。
4. 刷新后数据保留。

### **（十二）交易安全提醒**
交易安全提醒作为展示型功能，用于增强项目真实感和用户体验。

主要功能包括：

1. 提醒学生尽量选择公共地点交易。
2. 提醒贵重物品注意验真。
3. 提醒保护个人联系方式和隐私。
4. 提醒警惕异常低价和异常交易要求。
5. 以卡片、提示框或公告栏形式展示。

## **六、核心业务流程**
### **（一）身份创建流程**
用户首次进入系统，填写本地身份信息，包括昵称、学院、校区和身份角色。系统保存用户信息后进入首页。后续发布信息、收藏信息、发送消息和个人中心展示均基于该本地用户信息完成。

流程如下：

用户进入系统 → 创建本地身份 → 保存用户信息 → 进入今日集市首页

### **（二）信息浏览流程**
用户进入集市信息浏览页，系统加载校园信息列表。用户可以通过关键词、类型、校区、状态等条件进行筛选，也可以按时间、热度或价格进行排序。用户点击某条信息后进入详情页。

流程如下：

进入集市浏览页 → 加载信息列表 → 搜索/筛选/排序 → 查看信息详情

### **（三）信息发布流程**
用户进入信息发布页，选择发布类型。系统根据不同类型显示对应字段。用户填写信息并通过表单校验后提交，数据写入 JSON Server，发布成功后列表页可看到新增信息。

流程如下：

进入发布页 → 选择信息类型 → 填写对应字段 → 表单校验 → 提交数据 → 发布成功 → 列表显示新信息

### **（四）收藏流程**
用户在列表页或详情页点击收藏按钮，系统记录收藏关系，并更新页面收藏状态。用户可以在个人中心查看收藏列表，也可以取消收藏。

流程如下：

查看信息 → 点击收藏 → 保存收藏记录 → 个人中心展示 → 取消收藏

### **（五）模拟砍价流程**
用户查看二手商品详情，输入出价并提交。系统根据出价生成模拟回复，同时将买家出价和卖家回复写入消息记录，并生成或更新会话。

流程如下：

进入二手商品详情 → 输入出价 → 生成砍价消息 → 生成模拟回复 → 更新消息中心

### **（六）消息沟通流程**
用户从信息详情进入消息沟通，系统展示与发布者的会话记录。用户发送消息后，系统追加聊天记录，并可生成模拟回复和未读提醒。

流程如下：

进入详情页 → 点击联系发布者 → 打开会话 → 发送消息 → 追加聊天记录 → 生成模拟回复

### **（七）状态更新流程**
信息发布者进入个人中心，查看我的发布，并根据实际情况更新信息状态。不同类型的信息可以使用不同状态，例如进行中、已完成、已关闭、已找回、已认领等。

流程如下：

进入个人中心 → 查看我的发布 → 选择信息 → 更新状态 → 列表和详情同步显示

### **（八）趋势看板流程**
用户进入趋势看板页，系统获取校园信息数据，前端根据类型、校区、状态等字段进行统计，并通过图表展示结果。

流程如下：

进入趋势看板 → 获取信息数据 → 前端统计计算 → ECharts 展示图表

## **七、页面需求分析**
本项目建议包含以下主要页面：

### **（一）身份创建页**
用于创建本地用户档案。页面包含昵称、学院、校区、身份角色等输入项，并提供开始使用按钮。

### **（二）今日集市首页**
用于展示系统概览和功能入口。页面包含用户欢迎信息、快捷入口、最新信息、热门信息、我的收藏、未读消息和基础统计卡片。

### **（三）集市信息列表页**
用于展示校园信息列表。页面包含搜索框、类型筛选、校区筛选、状态筛选、排序控件和信息卡片列表。

### **（四）信息详情页或详情抽屉**
用于展示单条信息完整内容。页面包含标题、图片、标签、状态、发布人、地点、描述、操作按钮等内容。二手交易类型应包含模拟砍价入口。

### **（五）信息发布页**
用于发布不同类型的校园信息。页面包含类型选择、通用字段、类型专属字段、表单校验和提交按钮。

### **（六）消息中心页**
用于展示会话列表和聊天记录。页面包含会话列表、消息内容区、输入框和发送按钮。

### **（七）个人中心页**
用于展示当前用户资料、我的发布、我的收藏和状态管理入口。

### **（八）趋势看板页**
用于展示数据统计图表。页面包含类型占比、校区分布、状态统计和热门信息排行等图表或卡片。

### **（九）安全提醒页或安全提醒组件**
用于展示校园线下交易和信息沟通的安全提示。该功能可以独立成页，也可以作为首页或详情页中的提示组件。

## **九、非功能需求**
### **（一）易用性需求**
系统界面应简洁清晰，主要功能入口明确。用户能够快速完成浏览、搜索、发布、收藏和消息沟通等操作。

### **（二）可维护性需求**
项目结构应清晰，页面、组件、API、Store、工具函数应合理拆分，避免所有代码集中在单个文件中。

### **（三）响应式需求**
页面应适配常见浏览器窗口大小，在桌面端具有良好展示效果。对于列表、卡片、表单和图表，应避免明显布局错乱。

### **（四）数据一致性需求**
通过 JSON Server 管理核心业务数据。页面刷新后，应能够重新从接口获取数据。Pinia 主要用于状态管理，不应把全部业务数据长期保存在 Pinia 中。

### **（五）可演示性需求**
项目应能够在本地环境稳定运行。学生应能够通过演示流程展示身份创建、信息发布、信息浏览、详情查看、收藏、消息、个人中心和趋势看板等功能。

### **（六）安全提示需求**
由于项目涉及校园线下交易和信息沟通，应在界面中提供必要的交易安全提醒，但不实现真实风控系统。

## **十、功能边界与不做内容**
为保证项目在实训周期内可完成，本项目明确以下功能边界：

1. 不实现真实注册、登录和密码认证。
2. 不实现真实用户权限控制。
3. 不实现真实支付功能。
4. 不实现订单系统。
5. 不实现物流配送。
6. 不实现平台审核。
7. 不实现实时聊天和 WebSocket。
8. 不实现复杂信用分计算。
9. 不实现推荐算法。
10. 不实现复杂后台管理系统。
11. 不实现真实图片上传，可使用预设图片或图片地址。
12. 不实现真实地图定位。
13. 不实现真实短信、邮箱或第三方通知。

模拟砍价、消息回复、信用分展示、趋势统计等功能均为教学模拟，用于训练前端交互、状态管理、数据处理和项目展示能力。

## **十二、需求分析总结**
“校园轻集市”的需求来源于大学生日常生活中的二手交易、失物招领、拼单搭子和跑腿委托四类高频场景。四类场景虽然业务含义不同，但都可以抽象为校园信息资源，并通过统一的信息发布、信息浏览、详情查看、收藏关注、消息沟通、状态更新和数据展示等功能进行承载。

本项目需求分析采用用户故事方式表达用户目标，用功能需求明确开发任务，用数据需求支撑接口设计，用验收标准控制实现深度。通过这种方式，学生能够理解软件项目并不是直接从页面开始，而是从真实需求出发，逐步转化为角色、场景、功能、数据和交互流程。

在实现过程中，项目应始终保持教学型前端实训定位。对于砍价、消息、信用分、趋势看板等功能，应强调其教学模拟属性，不追求真实产品级复杂度。项目最终目标是帮助学生在 AI 辅助开发环境下，完成一个可运行、可演示、可答辩的前端工程实践项目。
